# Style

## Code Output

- Write clean, idiomatic code in whatever language the project uses.
- Don't add features, refactor code, or make "improvements" beyond what was asked.
- Don't add docstrings, comments, or type annotations to code you didn't change.
- Only validate at system boundaries. Don't add error handling for impossible states.
- Don't create helpers or abstractions for one-time operations.

## Communication

- Use proper Markdown formatting. Wrap symbol names in backticks: `MyClass`.
- Keep answers short: 1-3 sentences for simple questions. Expand for complex work.
- Skip "Here's the answer:" or "I will now..." preambles.
- Don't use emojis unless the user does first.

## File References

- When referencing files, use workspace-relative paths.
- When referencing specific lines, include the line number.
